Build a production-ready, multi-tenant white-label Status Page & Uptime Monitor SaaS platform with microservice architecture. Below is the complete specification:

---
##
## PROJECT OVERVIEW

A SaaS platform where businesses (tenants) sign up, configure their brand, add services to monitor, and get a fully white-labeled public status page on their own custom domain (e.g. status.theirdomain.com). Their end users visit that domain and see real-time service health, incidents, and can subscribe to alerts.

---

## ARCHITECTURE

Microservice architecture with two independently deployable services:
- backend/ → FastAPI (Python)
- frontend/ → Next.js 14 (App Router)
- caddy/ → Reverse proxy handling custom domains
- Shared infrastructure: PostgreSQL, Redis
- Each service has its own Dockerfile, Jenkinsfile, and values.yaml
- docker-compose.yml for local development only

## BACKEND — FastAPI (Python)

### Tech
- FastAPI with async/await throughout
- SQLAlchemy 2.0 async ORM
- Alembic for migrations
- PostgreSQL as primary database
- Redis for pub/sub and caching
- APScheduler for background uptime check jobs
- Pydantic v2 for schemas
- JWT auth with python-jose
- httpx for async HTTP health checks
- Ruff for linting

### Database Models

**Tenant**
- id (UUID)
- name
- slug (unique, used for subdomain fallback)
- custom_domain (nullable, e.g. status.acmecloud.com)
- logo_url
- brand_color (hex)
- owner_email
- created_at

**Monitor**
- id (UUID)
- tenant_id (FK)
- name
- url
- check_interval_seconds (60, 300, 600)
- status (operational, degraded, outage)
- last_checked_at
- created_at

**UptimeLog**
- id (UUID)
- monitor_id (FK)
- checked_at
- response_time_ms
- status_code
- is_up (boolean)

**Incident**
- id (UUID)
- tenant_id (FK)
- monitor_id (FK, nullable)
- title
- message
- severity (minor, major, critical)
- status (investigating, identified, monitoring, resolved)
- created_at
- resolved_at

**Subscriber**
- id (UUID)
- tenant_id (FK)
- email
- confirmed (boolean)
- token (for confirmation/unsubscribe)
- created_at

### API Routes

**Auth**
- POST /api/auth/register
- POST /api/auth/login
- GET /api/auth/me

**Tenants (authenticated)**
- GET /api/tenants/me
- PUT /api/tenants/me (update brand, logo, custom_domain)

**Monitors (authenticated)**
- GET /api/monitors
- POST /api/monitors
- PUT /api/monitors/{id}
- DELETE /api/monitors/{id}
- GET /api/monitors/{id}/uptime?days=30

**Incidents (authenticated)**
- GET /api/incidents
- POST /api/incidents
- PUT /api/incidents/{id}
- DELETE /api/incidents/{id}

**Subscribers**
- POST /api/subscribers (public, email signup)
- GET /api/subscribers/confirm/{token}
- GET /api/subscribers/unsubscribe/{token}

**Public (no auth, used by frontend status page)**
- GET /api/public/{tenant_slug_or_domain} → returns tenant brand + monitor statuses + active incidents
- GET /api/public/{tenant_slug_or_domain}/history?days=90
- WebSocket /api/ws/{tenant_slug_or_domain} → real-time status updates

### Monitor Worker
- APScheduler runs background jobs per monitor based on check_interval_seconds
- On each check: ping the URL with httpx, record response time and status in UptimeLog
- If monitor goes from up → down: auto-create incident, publish to Redis channel, send email to tenant owner + all confirmed subscribers
- If monitor goes from down → up: auto-resolve incident, publish recovery to Redis, send recovery email
- WebSocket handler subscribes to Redis channel and pushes updates to connected clients

### Email Notifications (SMTP)
- Incident created: notify tenant owner + subscribers
- Incident resolved: notify tenant owner + subscribers
- New subscriber confirmation email with confirm link
- Use Jinja2 templates for emails

## CADDY — White Label Custom Domains

### Caddyfile
{
admin off
auto_https on
}
Platform dashboard and API
app.yourplatform.com {
reverse_proxy frontend:3000
}
api.yourplatform.com {
reverse_proxy backend:8000
}
Wildcard for tenant custom subdomains
:443 {
tls {
on_demand
}
reverse_proxy frontend:3000
}

- Caddy handles TLS automatically via Let's Encrypt for every custom domain
- On-demand TLS means when status.acmecloud.com first hits Caddy, it auto-provisions a cert
- Frontend detects the hostname and passes it to the API to resolve the correct tenant
- No manual SSL setup needed per tenant

## IMPORTANT IMPLEMENTATION NOTES

1. Multi-tenancy resolution: When a request hits the frontend on a custom domain (e.g. status.acmecloud.com), Next.js middleware reads the hostname from the request headers and passes it to the backend public API. The backend resolves the tenant by matching custom_domain field first, then falls back to slug.

2. Real-time: Use native WebSocket in FastAPI. On monitor status change, publish to Redis channel keyed by tenant_id. WebSocket handler subscribes to that channel and broadcasts to all connected clients on that tenant's status page.

3. Uptime calculation: For each monitor, calculate uptime % per day by counting (up_checks / total_checks) * 100 for each day. Store this in UptimeLog table and aggregate on query.

4. Brand color application: The public status page receives brand_color from the API and applies it as a CSS variable (--brand-color) in the layout. All accent elements (header, badges, buttons) use this variable.

5. On-demand TLS in Caddy: Configure Caddy with on_demand_tls so it auto-provisions Let's Encrypt certs for any custom domain that points to your server. Add an ask endpoint on your backend that Caddy calls to verify the domain belongs to a registered tenant before issuing a cert.

6. Health endpoints: Add GET /health on backend returning {"status": "ok"}. Next.js frontend should have a /api/health route as well. These are used by Kubernetes liveness and readiness probes defined in values.yaml.

7. Database migrations: Alembic is configured with async SQLAlchemy. Run migrations as an init container or startup step before the FastAPI app starts.

8. All API responses must be consistent: {"data": ..., "message": ..., "success": true/false}

9. Use environment variables for all secrets — never hardcode. All secrets injected via Kubernetes secrets referenced in values.yaml.

10. The Jenkinsfile stages call shared library functions: dockerBuild(), trivyScan(), dockerPush(), helmDeploy(), slackNotify(). Implement these as stubs with comments if the shared library is external.
